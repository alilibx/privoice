## Problem / goal

Privoice is on-device-only today. The first web slice must stand up the cloud
spine: a **React + Vite SPA on Convex** where a user can **sign up / log in / log
out** (email + password) and land on an authenticated **dashboard that lists and
creates their meetings** (title + notes only â€” no audio/STT yet). Every backend
call is authorized by identity; a user only ever sees their own data.

This proves, end to end: Convex backend + Convex Auth + reactive per-user data +
the Vite app shell + the calm-teal design language on web â€” the foundation every
later O-slice (billing, AI proxy, chat-with-docs, mobile online client) builds on.

Prior decisions this rests on: **O0 is GO** (Flutterâ†”Convex de-risked; web uses
the official `convex/react` client). Stack: **React + Vite SPA**, no Next.js;
Convex is the full backend; heavy work later runs in Convex Node actions.

## Security & privacy (first-class â€” per standing directive)

Security and privacy are the top priority for Privoice, explicitly extended to the
online tier. O1 sets the patterns every later slice inherits:

- **Identity on every call.** Each query/mutation resolves the caller with
  `getAuthUserId(ctx)` and **throws if unauthenticated**. There is no unauthenticated
  data path. Meetings are keyed by `userId`; reads/writes/deletes are scoped to the
  caller. **Tested for isolation**, not assumed (a second user sees zero rows; a
  cross-user `remove` is refused).
- **No secrets in code or logs.** The deployment URL lives in `.env.local`
  (gitignored) as `VITE_CONVEX_URL`. Auth signing keys (`JWT_PRIVATE_KEY`/JWKS) are
  set as **Convex deployment env vars** by the auth setup â€” never committed, never
  logged, never printed. The agent does not handle these secrets; the user runs the
  account-bound setup/deploy commands.
- **Passwords** are handled entirely by Convex Auth's Password provider (hashing/
  storage server-side in Convex); we never store or log raw passwords. No password
  ever transits our own logging.
- **Minimal data.** O1 stores only `title`, optional `notes`, `createdAt`, `status`,
  `userId`. No audio/transcript/PII beyond the account email Convex Auth holds.
- **Online is opt-in by nature** (web is inherently online); on mobile the online
  tier stays off by default (later slices). Nothing here changes the on-device
  default or its zero-network gate.
- **A `/security-review` pass is part of O1's definition of done** before merge.

## Convex backend

**Auth (`convex/auth.ts`, `http.ts`, `auth.config.ts`):** `@convex-dev/auth` with
the **Password** provider only (email + password). `http.ts` mounts the auth routes
via `auth.addHttpRoutes(http)`. Setup (`npx @convex-dev/auth`) generates
`auth.config.ts` and provisions the signing keys as deployment env vars.

**Schema (`convex/schema.ts`):**
```ts
import { defineSchema, defineTable } from "convex/server";
import { authTables } from "@convex-dev/auth/server";
import { v } from "convex/values";

export default defineSchema({
  ...authTables, // users, authSessions, authAccounts, â€¦
  meetings: defineTable({
    userId: v.id("users"),
    title: v.string(),
    notes: v.optional(v.string()),
    createdAt: v.number(),
    status: v.string(), // "note" for O1 (no audio/STT yet); mirrors mobile MeetingStatus naming
  }).index("by_user", ["userId"]),
});
```
Field names deliberately mirror the mobile `Meeting` (`title`, `createdAt`,
`status`) for later parity; `transcript`/`minutes`/`actionItems` are **not** modeled
yet (they arrive with audio-upload/STT slices, whose real shape isn't known yet â€”
YAGNI).

**Functions (`convex/meetings.ts`)** â€” each resolves `const userId = await
getAuthUserId(ctx)` and throws (`ConvexError("Not authenticated")`) when null:
- `list` (query): `ctx.db.query("meetings").withIndex("by_user", q => q.eq("userId", userId)).order("desc").collect()` â†’ the caller's meetings, newest first.
- `create` (mutation, args `{ title: v.string(), notes: v.optional(v.string()) }`):
  validates non-empty `title`; inserts `{ userId, title, notes, createdAt: Date.now(), status: "note" }`; returns the new id.
- `remove` (mutation, args `{ id: v.id("meetings") }`): loads the row, throws if
  missing or `row.userId !== userId`, else deletes. (No cross-user deletion.)

## Web app (`apps/web/src`)

- `main.tsx`: `ConvexReactClient(import.meta.env.VITE_CONVEX_URL)` wrapped in
  `ConvexAuthProvider`.
- `App.tsx`: `convex/react` auth-state components â€”
  `<AuthLoading>` â†’ spinner; `<Unauthenticated>` â†’ `<AuthForm/>`;
  `<Authenticated>` â†’ `<Dashboard/>`. No router in O1.
- `AuthForm.tsx`: email + password fields, a sign-in/sign-up toggle, submit via
  `useAuthActions().signIn("password", { email, password, flow })`. Inline error
  display; disabled/loading state on submit.
- `Dashboard.tsx`: shows the signed-in email + a **Sign out** button
  (`useAuthActions().signOut`); `useQuery(api.meetings.list)` renders the live list;
  a **New meeting** form (`useMutation(api.meetings.create)`, title required, notes
  optional); each row has a delete (`api.meetings.remove`). Reactive â€” a create/
  delete updates the list instantly.
- **Design:** Tailwind with **calm-teal tokens** mirrored from the mobile theme
  (primary `#12708D`; tinted-neutral greys; soft radii; light + dark via
  `prefers-color-scheme`). `theme.ts` / CSS vars hold the palette so it's one source
  of truth. The web must read as the same product, not a generic starter.

## Testing & verification

- **Backend (`convex-test` + vitest)** â€” the security-critical layer:
  - `create` then `list` returns exactly the caller's row(s).
  - **Isolation:** seed two identities; user A's `list` never returns user B's rows.
  - `remove` deletes the caller's own row; **refuses** another user's row (throws).
  - Unauthenticated `list`/`create`/`remove` **throw**.
- **Web (vitest + @testing-library/react):** `AuthForm` renders its fields when
  unauthenticated; a light `Dashboard` render with a mocked Convex client shows the
  list + New-meeting form. (Auth-state components mocked; no live network.)
- **Manual e2e against the live deployment:** sign up â†’ create a meeting â†’ see it
  appear live â†’ sign out â†’ list gone; a second account sees none of the first's data.
- **Security review:** run `/security-review` on the O1 diff before merge; confirm
  no secret is committed/logged, every function is identity-gated, and there is no
  unauthenticated data path.

## Account-bound steps (user runs; agent won't handle secrets)

1. `cd apps/web && npm install`
2. `npx @convex-dev/auth` â€” provisions auth signing keys as deployment env vars,
   writes `auth.config.ts` (links to the existing `colorless-mammoth-659` deployment).
3. `npx convex dev` (or `--once`) â€” codegen + push functions.
The agent scaffolds all code and provides these exact commands; the user runs the
CLI/secret-provisioning steps. Headless verification (`convex-test`/vitest) and web
component tests run without secrets.
